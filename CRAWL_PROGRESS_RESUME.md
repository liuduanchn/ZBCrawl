# 招标信息爬取断点续传功能说明

## 功能概述

招标信息爬取现在支持断点续传功能，当网络错误导致爬取中断时，下次可以从中断的位置继续爬取，无需从头开始。

## 核心特性

### 1. 自动进度记录
- 爬取过程中自动记录当前进度
- 包括：当前爬取的学校索引、已完成的学校数、失败的学校数、总爬取数量
- 每处理完一个学校后自动更新进度

### 2. 中断检测与恢复
- 检测到未完成的爬取任务时自动提示
- 显示上次中断的位置和统计信息
- 点击"开始爬取"自动从断点继续

### 3. 进度管理
- **查看进度**：页面加载时自动显示当前爬取进度
- **重置进度**：可以手动清除进度记录，从头开始爬取
- **完成标记**：爬取完成后自动标记为已完成状态

## 使用说明

### 正常爬取流程

1. **开始爬取**
   - 点击"开始爬取"按钮
   - 系统自动创建进度记录
   - 实时显示爬取进度和统计

2. **监控进度**
   - 查看爬取进度条
   - 查看实时日志
   - 查看已爬取的学校数量

3. **爬取完成**
   - 显示完成统计信息
   - 自动标记进度为"已完成"

### 断点续传流程

#### 场景1：网络中断后继续爬取

1. **中断发生**
   - 网络错误导致爬取中断
   - 进度自动保存，状态标记为"running"

2. **检测未完成任务**
   - 重新打开页面时显示警告提示
   - 显示上次中断位置和统计信息：
     - 已完成学校数
     - 失败学校数
     - 已爬取招标信息数

3. **继续爬取**
   - 点击"开始爬取"按钮
   - 系统自动从上次中断的位置继续
   - 提示："检测到未完成的爬取任务，从第 X 个学校继续..."

#### 场景2：爬取完成后重新爬取

1. **检测已完成任务**
   - 如果上次爬取已完成，显示提示信息
   - 显示上次爬取的统计数据

2. **重置进度**
   - 点击"重置进度"按钮
   - 确认后清除进度记录
   - 下次爬取将从头开始

## API 接口

### 1. 获取爬取进度

**接口**：`GET /api/tenders/crawl`

**响应**：
```json
{
  "success": true,
  "data": {
    "id": "xxx",
    "crawlType": "tenders",
    "schoolName": "某学校",
    "currentIndex": 50,
    "totalSchools": 1000,
    "status": "running",
    "completedSchools": 50,
    "failedSchools": 2,
    "totalCount": 500,
    "errorMessage": null,
    "lastCrawledAt": "2026-01-22T10:00:00Z",
    "createdAt": "2026-01-22T09:00:00Z",
    "updatedAt": "2026-01-22T10:00:00Z"
  }
}
```

### 2. 开始/继续爬取

**接口**：`POST /api/tenders/crawl`

**响应**：SSE 流式输出

**事件类型**：
- `start`: 开始爬取
- `resume`: 从断点继续
- `warning`: 检测到已完成任务
- `progress`: 爬取进度更新
- `tender`: 新增招标信息
- `success`: 学校爬取成功
- `error`: 学校爬取失败
- `complete`: 爬取完成
- `heartbeat`: 心跳保活

**恢复事件示例**：
```json
{
  "type": "resume",
  "message": "检测到未完成的爬取任务，从第 51 个学校继续...",
  "startIndex": 50,
  "completedSchools": 50,
  "failedSchools": 2,
  "totalCount": 500
}
```

### 3. 重置爬取进度

**接口**：`DELETE /api/tenders/crawl`

**响应**：
```json
{
  "success": true,
  "message": "爬取进度已重置"
}
```

## 数据库表结构

### crawl_progress 表

| 字段 | 类型 | 说明 |
|------|------|------|
| id | VARCHAR(36) | 主键 |
| crawl_type | VARCHAR(50) | 爬取类型（tenders） |
| school_name | VARCHAR(255) | 当前爬取的学校名称 |
| current_index | INTEGER | 当前索引（学校数组中的位置） |
| total_schools | INTEGER | 总学校数 |
| status | VARCHAR(20) | 状态：pending/running/completed/failed |
| completed_schools | INTEGER | 已完成的学校数 |
| failed_schools | INTEGER | 失败的学校数 |
| total_count | INTEGER | 总爬取数量 |
| error_message | TEXT | 错误信息 |
| last_crawled_at | TIMESTAMP | 最后爬取时间 |
| created_at | TIMESTAMP | 创建时间 |
| updated_at | TIMESTAMP | 更新时间 |

## 状态说明

### pending
- 等待中状态
- 进度已创建但尚未开始爬取

### running
- 进行中状态
- 正在爬取，可以从中断位置继续

### completed
- 已完成状态
- 所有学校已爬取完成
- 可以重置后重新开始

### failed
- 失败状态
- 爬取过程中发生严重错误
- 需要手动重置后重新开始

## 前端界面说明

### 进度提示区域

当检测到未完成的爬取任务时，会显示黄色的警告提示框，包含：
- 标题："⚠️ 检测到未完成的爬取任务"
- 中断信息：上次爬取在第 X 个学校（学校名称）时中断
- 统计数据：
  - 已完成学校数
  - 失败学校数
  - 已爬取招标信息数
- 操作提示：点击"开始爬取"按钮将继续从上次中断的位置继续爬取

### 按钮状态

- **⚠️ 有未完成的爬取任务**：提示按钮（不可点击）
- **重置进度**：当爬取完成后出现，用于清除进度记录
- **开始爬取**：正常爬取或继续爬取
- **停止爬取**：中断当前爬取

## 注意事项

1. **进度自动保存**
   - 每处理完一个学校后自动保存进度
   - 无需手动保存

2. **中断处理**
   - 网络错误、超时等会触发断点续传
   - 手动停止爬取也会保存进度

3. **并发控制**
   - 同一时间只能有一个爬取任务在运行
   - 新的爬取会继续未完成的任务

4. **重置谨慎**
   - 重置进度会删除所有进度记录
   - 已爬取的招标信息不会删除
   - 如需删除数据，请使用"清空全部"功能

## 常见问题

### Q: 为什么显示"有未完成的爬取任务"？
A: 上次爬取过程中断，系统已保存进度。点击"开始爬取"可以继续。

### Q: 如何从头开始重新爬取？
A: 如果上次爬取已完成，会显示"重置进度"按钮。点击后清除进度记录，下次爬取将从头开始。

### Q: 断点续传会重复爬取吗？
A: 不会。系统会从中断的学校继续，已爬取的学校不会重复。

### Q: 爬取失败的学校会重试吗？
A: 不会自动重试。需要重置进度后重新爬取。

### Q: 如何清除已爬取的招标信息？
A: 使用"清空该学校数据"或"清空全部"按钮。

## 技术实现细节

### 进度更新策略
- 每处理完一个学校后立即更新数据库
- 使用乐观更新确保数据一致性
- 更新失败不影响爬取流程

### 中断检测
- 通过检查 status 字段判断是否需要断点续传
- running 状态表示未完成的任务
- completed 状态表示已完成的任务

### 错误处理
- 单个学校失败不影响整体流程
- 记录失败的学校数量
- 严重错误（如数据库连接失败）会标记为 failed 状态

### 性能优化
- 使用索引加速查询
- 批量更新减少数据库操作
- 流式输出减少内存占用
