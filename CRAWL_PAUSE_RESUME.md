# 招标信息爬取暂停功能说明

## 功能概述

招标信息爬取现在支持手动暂停和自动暂停功能，可以在爬取过程中暂停，并从暂停位置恢复爬取。

## 核心特性

### 1. 手动暂停
- 点击"暂停爬取"按钮暂停当前爬取
- 系统会完成当前学校的爬取后停止
- 保存当前进度和状态

### 2. 自动暂停
- 单次爬取时长超过 15 分钟后自动暂停
- 防止长时间占用服务器资源
- 保存当前进度，可随时恢复

### 3. 恢复爬取
- 点击"恢复爬取"按钮从暂停位置继续
- 无需从头开始
- 继续爬取剩余的学校

## 使用说明

### 手动暂停流程

1. **开始爬取**
   - 点击"开始爬取"按钮
   - 系统开始爬取学校数据

2. **暂停爬取**
   - 在爬取过程中点击"暂停爬取"按钮
   - 系统发送暂停指令
   - 等待当前学校爬取完成
   - 自动保存进度并停止

3. **查看暂停状态**
   - 页面显示"爬取已暂停"
   - 显示暂停原因
   - 显示已完成的统计数据

4. **恢复爬取**
   - 点击"恢复爬取"按钮
   - 系统从暂停位置继续
   - 无需重新开始

### 自动暂停流程

1. **自动触发**
   - 单次爬取时长超过 15 分钟
   - 系统自动发送暂停指令
   - 保存当前进度

2. **查看自动暂停**
   - 页面显示"爬取已自动暂停"
   - 显示暂停原因：单次爬取时长超过 15 分钟
   - 显示已完成的统计数据

3. **恢复爬取**
   - 点击"恢复爬取"按钮
   - 系统从暂停位置继续
   - 计时器重置，可继续爬取 15 分钟

## 按钮说明

### 开始爬取
- **状态**：未爬取时显示
- **功能**：开始新的爬取任务
- **禁用条件**：正在爬取或已暂停时

### 暂停爬取
- **状态**：正在爬取时显示
- **功能**：暂停当前爬取
- **行为**：完成当前学校后停止

### 停止爬取
- **状态**：正在爬取时显示
- **功能**：完全停止爬取
- **行为**：立即停止，不保存状态

### 恢复爬取
- **状态**：已暂停时显示
- **功能**：从暂停位置继续
- **行为**：重置计时器，继续爬取

## 状态管理

### 状态列表
- `pending`: 等待中
- `running`: 进行中
- `paused`: 已暂停
- `completed`: 已完成
- `failed`: 失败

### 状态转换
```
pending -> running
running -> paused (手动暂停)
running -> paused (自动超时)
paused -> running (恢复)
running -> completed
running -> failed
```

## API 接口

### 暂停爬取

**接口**：`POST /api/tenders/crawl/pause`

**请求**：无

**响应**：
```json
{
  "success": true,
  "message": "暂停指令已发送，等待当前学校爬取完成..."
}
```

**错误响应**：
```json
{
  "success": false,
  "error": "没有正在进行的爬取任务"
}
```

### 获取暂停状态

**接口**：`GET /api/tenders/crawl/pause`

**响应**：
```json
{
  "success": true,
  "paused": false,
  "status": "running"
}
```

### 恢复爬取

**接口**：`DELETE /api/tenders/crawl/pause`

**响应**：
```json
{
  "success": true,
  "message": "爬取已恢复"
}
```

## 超时配置

### 当前配置
- 超时时间：15 分钟
- 超时单位：毫秒（900000）
- 配置位置：`src/app/api/tenders/crawl/route.ts`

### 修改超时时间
```typescript
// 修改 MAX_CRAWL_DURATION 的值
const MAX_CRAWL_DURATION = 15 * 60 * 1000; // 15分钟
// 改为 30 分钟
const MAX_CRAWL_DURATION = 30 * 60 * 1000; // 30分钟
```

## 注意事项

1. **暂停时机**
   - 暂停指令发送后，系统会完成当前学校的爬取
   - 等待时间取决于当前学校的处理速度
   - 通常需要等待 10-30 秒

2. **超时自动暂停**
   - 超时后会自动暂停，无需人工干预
   - 保存当前进度，可随时恢复
   - 恢复后计时器重置，可继续爬取

3. **并发控制**
   - 同一时间只能有一个爬取任务
   - 暂停状态下不能开始新的爬取
   - 必须先恢复或停止当前任务

4. **进度保存**
   - 暂停时自动保存进度
   - 包括：当前索引、已完成的学校数、总爬取数
   - 恢复时从暂停位置继续

5. **网络中断**
   - 网络中断后需要重新开始爬取
   - 系统会自动检测并从断点继续
   - 与手动暂停的处理方式相同

## 常见问题

### Q: 点击暂停后为什么没有立即停止？
A: 暂停指令会等待当前学校的爬取完成后才停止，以确保数据完整性。

### Q: 超时时间可以修改吗？
A: 可以。修改 `src/app/api/tenders/crawl/route.ts` 中的 `MAX_CRAWL_DURATION` 常量。

### Q: 暂停后恢复会从头开始吗？
A: 不会。系统会从暂停的位置继续，已爬取的数据不会丢失。

### Q: 自动暂停后可以手动恢复吗？
A: 可以。点击"恢复爬取"按钮即可，与手动暂停后的恢复方式相同。

### Q: 暂停状态会保持多久？
A: 暂停状态会永久保存在数据库中，直到手动恢复或重置进度。

### Q: 如何查看暂停原因？
A: 在爬取进度信息区域会显示暂停原因。

### Q: 暂停期间服务器资源会被占用吗？
A: 不会。暂停后会释放所有资源，只保留进度数据。

## 技术实现细节

### 暂停检测
- 从数据库读取当前爬取进度状态
- 每处理一个学校前检查一次
- 发现暂停状态立即停止

### 超时检测
- 记录爬取开始时间
- 每处理一个学校前检查一次
- 超过最大时长自动暂停

### 状态同步
- 使用数据库存储暂停状态
- 前端定期查询暂停状态
- 确保状态一致性

### 进度恢复
- 恢复时重置数据库状态
- 清除暂停标记
- 从 currentIndex 继续爬取
